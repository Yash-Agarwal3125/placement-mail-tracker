## Additional load-bearing facts (not in the assumption list)

- **Sheets writes are clear-then-write per sync** (`sheets_sync.py:318-353`) — the sheet has no incremental diff. The calendar sync will be the *first* incremental/diff-based render target; "same philosophy as sheets sync" holds for *DB is source of truth* but **not** for the write mechanism.
- **Six tabs exist**: ACTION REQUIRED, MY APPLICATIONS, ALL DRIVES, UPCOMING EVENTS, Company History, RECENT UPDATES + Dashboard (`sheets_sync.py:293-301`).
- `fetch_active_drives_only()` (`manager.py:544-566`) is the "active set" the sheets sync consumes (status='active' AND current_status in OPEN…SELECTED) — the natural input for `derive.py` and for the partial-fetch guard ("zero active drives → abort stale pass").
- `main.py` has **no argparse today** (`main.py:30, 157-158`) — `--calendar-dry-run` / `--calendar-rebuild` introduce the first CLI flags.
- `googleapiclient` is already a dependency (`requirements.txt:4-6`); the Calendar v3 API needs **no new package**.
- Gemini extraction results are validated by Pydantic `PlacementExtraction` (`pmt/ai/models.py:10`), but date fields remain free strings end-to-end.
- `drive_id` uniqueness is best-effort (`LIKE`-count suffixing at `manager.py:822-828`), not DB-enforced; `UNIQUE(drive_id, event_type)` on `calendar_events` is still safe because `calendar_events` rows are only minted from existing `opportunities` rows, but FK-by-`opportunities.id` is the more robust identity anchor.
